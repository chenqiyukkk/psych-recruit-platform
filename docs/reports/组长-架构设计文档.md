# 软件架构设计文档（心理学被试招募平台）

> 说明：本文档为 Markdown 草稿版，供组内同步与后续导出 PDF 使用。  
> 技术栈：后端 Spring Boot 2.7；Web 前端 Vue3 + Element Plus；小程序 微信原生；数据库 MySQL 8.0；缓存 Redis。  
> 关键特性：被试匿名化、线下支付确认、双向评价、申诉机制。

## 1. 系统概述

心理学被试招募平台面向三类角色：
- 被试：完善匿名档案、报名实验、签到、确认收款、评价研究者、发起申诉
- 研究者：创建/发布实验、筛选与审核报名、确认支付、评价被试、处理支付争议（配合管理员）
- 管理员：用户与内容治理、申诉审核、违规处理

系统目标：
- 提供实验发布与报名匹配能力
- 保障被试隐私（匿名化）
- 提供线下支付的“双方确认”闭环
- 通过双向评价与信誉分机制降低失约与纠纷
- 通过申诉机制保障纠错与可追溯性

## 2. 整体架构

### 2.1 分层说明

- 表现层（Clients）
  - Web 管理端（研究者/管理员）：实验管理、报名审核、统计、申诉处理等
  - 微信小程序（被试端为主）：浏览/报名/签到/支付确认/评价等
- 接入层（API Gateway 可选）
  - 统一 API（REST），统一鉴权（JWT）
- 业务层（Backend: Spring Boot）
  - 用户与权限模块
  - 实验模块
  - 报名/签到模块
  - 支付确认模块（线下）
  - 评价与信誉分模块
  - 申诉模块
  - 通知/消息模块
- 数据层
  - MySQL：业务主数据
  - Redis：缓存、限流、验证码/临时状态（可选）
  - 文件存储：收款码、支付截图等（本地/对象存储可替换）

## 3. 技术选型及理由

### 3.1 后端：Spring Boot 2.7
- 成熟稳定、生态完善（JPA/Security/Validation）
- 易于分层与模块化开发，便于组员并行协作
- 便于与 Swagger/OpenAPI 集成，提升联调效率

### 3.2 Web 前端：Vue3 + Element Plus
- 组件生态成熟，适合后台管理系统快速搭建
- 与 REST API 对接简单，便于分页/表格/表单等典型场景

### 3.3 小程序：微信原生
- 覆盖被试端常用场景（报名、签到、确认收款、评价）
- 微信生态便于触达与使用

### 3.4 数据库：MySQL 8.0
- 事务与外键支持完善，适合强一致业务（报名、支付确认）
- JSON 字段可承载筛选条件/证据列表等半结构化数据

### 3.5 缓存：Redis（可选）
- 适用于验证码、限流、会话/临时状态、热点数据缓存
- 可扩展用于“互斥标签冷却期”查询加速（按需）

## 4. 关键设计决策

### 4.1 被试匿名化设计

核心原则：
- 被试对研究者侧尽量只暴露匿名信息（例如 `anonymous_id`、基础画像维度）
- 真实身份信息（手机号/邮箱）仅平台必要时可见（管理员权限），业务默认不向研究者直接暴露

数据设计要点：
- `users`：账号与角色、信誉分等
- `participant_profiles`：被试档案，含 `anonymous_id`（如 `P-2025-001`），承载年龄段/性别/专业大类/惯用手等

### 4.2 线下支付流程设计（双向确认）

目标：线下转账不可自动验真，因此用“双方确认 + 争议”实现闭环。

状态流转：
- `PENDING`：记录已创建，等待付款方确认
- `PAID`：付款方（研究者）确认已支付（可附支付截图）
- `CONFIRMED`：收款方（被试）确认已收款
- `DISPUTED`：任一方发起争议，进入申诉/审核流程

关键表：
- `payment_codes`：用户收款码（微信/支付宝）
- `payment_records`：与报名记录绑定的支付确认记录（以 `registration_id` 唯一约束）

### 4.3 双向评价机制设计

目标：提升匹配质量，降低失约率。

设计要点：
- `reviews` 支持两类评价方向：
  - `SUBJECT_TO_RESEARCHER`（被试评价研究者）
  - `RESEARCHER_TO_SUBJECT`（研究者评价被试）
- 总分 + 维度分（沟通/专业/守时）
- 支持匿名评价（`is_anonymous`）

### 4.4 申诉处理流程设计

目标：给出纠错渠道与审计链路。

设计要点：
- `appeals`：申诉记录，`appeal_type` 覆盖信誉分扣减/低评分/支付争议
- `evidence_urls`（JSON）：证据列表（截图等）
- `status`：`PENDING` → `UNDER_REVIEW` → `APPROVED/REJECTED`
- 审核人（管理员）与审核意见、审核时间可追溯

## 5. 安全性设计

### 5.1 JWT 认证
- 登录成功后签发 JWT
- 客户端在 `Authorization: Bearer <token>` 头中携带
- 后端解析 token，提取用户名与角色用于鉴权

### 5.2 密码加密
- 注册/修改密码时使用 BCrypt 哈希存储
- 后端不存储明文密码

### 5.3 权限控制
- 角色级别访问控制（被试/研究者/管理员）
- 资源级别控制：
  - 实验：仅创建者（研究者）或管理员可编辑/发布/删除
  - 支付记录：仅付款方/收款方/管理员可访问

### 5.4 传输安全
- 部署环境建议启用 HTTPS（由 Nginx 终止 TLS）
- 日志中避免打印敏感信息（token、密码、证据链接中的隐私字段等）

## 6. 隐私保护设计

- 匿名化：研究者侧默认不暴露被试真实身份字段
- 访问控制：管理员权限最小化、审计可追踪
- 数据最小化：仅采集业务必需的档案维度
- 知情同意：`consent_records` 记录签署版本、签署时间与必要的审计字段

## 7. 可扩展性与演进

- 后端按模块拆分包结构，便于后续拆成多服务（可选）
- 业务状态机（实验状态、支付状态、申诉状态）集中维护常量，便于扩展
- 文件存储抽象：本地路径可替换为 OSS/MinIO
